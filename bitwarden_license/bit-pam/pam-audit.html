The PAM Audit Log

How the synthesized access trail works, where it falls short of a compliance-grade audit record, and why privileged-access governance still needs an audit table of its own.

1The short version

Privileged Access Management (PAM) lets a member request temporary access to a protected credential, have it approved, and hold it for a fixed window. Governance teams need to answer, after the fact: who asked for access to what, who decided, who opened the secret, and when did it end? That record is the audit log.

A dedicated, purpose-built audit table for PAM was proposed and declined, not on technical grounds but because a new database table is only welcome if it is generic enough to serve many purposes, and there is already a team that owns the general-purpose event log. So we built the audit trail the only way left open to us: we synthesize it at query time from data the PAM entities already keep. No new table, no stored audit records.

The bottom line

The synthesized log is a genuinely clever compromise, good enough to show the trail in the product today. But it is a projection of live business data, not an independent record, and that gap is structural: the properties an auditor expects can’t be bolted onto a projection, only written down. That write is the thing we were told not to build. The case for building it, specifically not as a “generic” store, is what follows.

2Open questions for discussion

The agenda for the conversation this document is meant to start:

On compliance posture

  • Which frameworks (SOC 2, ISO 27001, HIPAA) does PAM need to satisfy, and do any of them require immutable / tamper-evident privileged-access logs? That answer decides whether the gap in Section 7 is a “nice to have” or a “must fix.”
  • Is best-effort logging acceptable for privileged access, or do we hold the line on “access fails if it can’t be logged”? The synthesized model can’t offer the latter.
  • Is credential-open auditing (who actually read the secret, and how often) required? It is the biggest thing synthesis cannot deliver.

On the store

  • Can we revisit the “generic table only” constraint given the Section 8 argument? Or, if a generic store is truly the only option, can it carry a PAM-specific extension/columns without becoming the event feed again?
  • What retention window and export formats (CSV, or standardized audit formats such as CEF / OCSF, per Section 9) does a PAM audit record need, independent of operational-data lifecycle?
  • Where does the boundary sit with the team that owns the existing event log? Parallel emission stands, but who owns the governance-grade store?

On scope & audience

  • Audience (decided): org-wide, gated by the audit-log permission (AccessEventLogs); anyone who can view the organization’s event logs sees the whole PAM trail, regardless of collection management.
  • Are the deferred kinds (credential access, kill-switch, freeze) in or out of the first compliant version?
Whatever we decide

None of this wastes the synthesized log: keep it as the in-product view. It just shouldn’t be mistaken for the record. And if a generic store is truly unavoidable, insist it carry PAM’s domain columns; otherwise we will have rebuilt the event log and solved nothing.

3Background: PAM and audit logs

For readers who don’t live in PAM. Skip to Section 4 if you do.

What PAM does

Some credentials in a vault are too sensitive to leave open to everyone who can see the collection. PAM gates them. When a member tries to open a gated credential, they don’t get the secret. Instead, they get a stripped-down view and a prompt to request access. If the request is approved, the member activates it when they’re ready, which mints a time-boxed lease. While the lease is live they can open the real credential; when it expires or is revoked, the gate closes again.

The cast

  • Requester: the member who wants access, and who later holds the lease.
  • Approver: who decides a request (approve or deny), or who ends someone’s lease early.
  • Governance viewer: anyone with the organization’s audit-log permission (AccessEventLogs), meaning org admins by default or a custom role granted it, who sees the whole organization’s trail. This is the audit log’s audience.

Plus the branches: a request can be cancelled or expire un-decided; an activation can be refused; a lease can be extended; and admins can create, edit, or delete the rules that gate collections, or hit an org-wide freeze / kill switch.

What an audit log is (and why it’s special)

An activity feed tells you what is happening. An audit log is a stronger promise: a durable, trustworthy record kept so that someone can later verify what happened, whether a security reviewer, an internal auditor, or a compliance framework (SOC 2, ISO 27001, HIPAA). For privileged access (the keys to the most sensitive data), that record is expected to be complete, immutable, tamper-evident, and guaranteed to exist. Hold onto those four words; Section 7 is about how a synthesized log measures up against them.

4The options on the table

Four designs were possible. One was built; one is what this document argues for.

Generic org event log Not the trail

The existing four-column activity feed (Timestamp, Client, Member, Event). It will carry PAM events (that requirement stands), but it collapses every event to one sentence, is gated behind an enterprise entitlement, is best-effort, and can only be queried by scope and date. It cannot be the audit trail; Section 9 explores why.

A generic audit table The political ask

A new table is acceptable only if it’s generic enough for many audit purposes, which is attractive organizationally. Its concrete form, a CosmosDB store sized for event-scale, is weighed in Section 10; why generic can’t serve PAM is Section 8.

Synthesized read-model What we built

No new table at all. Reconstruct the trail on demand from the state the PAM entities already keep. Ships today, adds no storage, and sidesteps the politics. But it carries the structural limits this document is about.

Dedicated PAM audit log The recommendation

A purpose-built, append-only, always-on store with PAM’s own columns and queries. The original research asked for this; the compromise we shipped is, ironically, the strongest evidence for why it’s needed.

5What we built: the synthesized log

The key move: there is no audit record anywhere. When a governance viewer opens the audit tab, the server reads the PAM entities that already exist (access requests, leases, and rules) and reconstructs a stream of audit events on the fly from the timestamps and actor fields those entities happen to retain (submitted_at, resolved_at, revoked_by, and so on). Each audit event is a derived value, never stored as a fact in its own right. It is a read-only view; nothing is ever written. That single choice is the source of everything good and everything limited about the approach.

The load-bearing idea: the synthesizability principle

An event can be reconstructed from entity state if and only if it left a durable, distinct, single-occurrence mark on a row that still exists.

Read that as four requirements. The mark must be durable (not overwritten), distinct (not blurred together with others), single-occurrence (the row can only remember one), and on a surviving row (not deleted). Break any one and the event simply cannot be synthesized; there is nothing to read it back from. Section 6 is the four ways that breaks.

6Where synthesis breaks

The synthesizability principle has a mirror image: four ways an event fails to leave a readable mark. Every one of them showed up in PAM, and each forced a compromise. The compromises are the point: each is a workaround for not having a place to write the record down.

1. Repeated events: one row can’t remember N times

Opening a leased credential can happen many times under one lease. A single lease row has one place to note “opened,” not one per open. The count, the times, the “did they actually read it” (the questions auditors care about most) have nowhere to live.

What we did: deferred it. Credential opened and access blocked contribute no events today. (This is the one place the plain event log, which appends a row per open, is actually better than synthesis.) The way to capture them without a dedicated table: a serialized list on the lease, one entry per open, instead of a single mark. That fills the coverage gap, but the per-open history rides on the mutable lease row, so it trades “capture less” for “bend the model further” rather than closing the compliance gap.

2. Overwrites: an edit destroys what came before

Editing a rule overwrites its previous settings and forgets who set them. The prior version is simply gone.

What we did: synthesize the latest edit only. “Rule updated” tells you the most recent change and who made it, never the history. (A lease extension escapes this only because PAM happens to preserve the old end date as a field on the extension request.)

3. Hard deletes: the row, and its whole story, vanish

Deleting a rule would remove the row, taking every mark with it. You couldn’t audit a rule that no longer exists, precisely the rule an investigator most wants to see.

What we did: changed delete to mean soft-delete: keep the row, stamp who deleted it and when, and hide it from every active read. That required a ripple of changes so a “deleted” rule stops gating access everywhere. A real behavior change, adopted to serve the audit trail.

4. Refused attempts: nothing happened, so nothing was stamped

When an activation is refused, the request by definition doesn’t change state. A trail built from state changes has nothing to read.

What we did: added a dedicated rejected_at field so a refusal leaves a mark. But a field holds one value, so it records only the last refusal; earlier attempts are lost.

Notice the pattern

Every fix above is either “capture less” (defer it, keep only the latest, keep only the last) or “bend the business model to preserve a mark” (soft-delete, an extra field). None of them is “write the event down when it happens,” because there is no audit store to write it to. We are spending real engineering effort to partially recover what a dedicated log would capture for free.

7Why “synthesized” may not be compliant

Recall the four words from Section 3: a privileged-access audit record is expected to be complete, immutable, tamper-evident, and guaranteed. Here is how a synthesized trail measures up, stated fairly, including where it does fine.

Where it holds up

Two genuine strengths

Always-on. Because the trail rides on PAM’s own data, it exists wherever PAM is sold; it is not gated behind the enterprise-events entitlement, unlike the org event log. Partial queryability. It can be filtered by collection, requester, approver, and kind, far better than a four-column sentence feed.

Where it doesn’t (and can’t)

It is not immutable

The “record” is the live business data. There is no separate, write-once copy. If a request, lease, or rule row is edited or removed in the normal course of business, the audit history derived from it changes or disappears with it. An audit log is supposed to be the one thing that doesn’t change after the fact; this one changes by construction.

It is not tamper-evident

Because there’s no independent record, there is nothing to compare against and no way to detect that history was altered. Tamper-evidence requires a second, protected copy of the truth. A projection is not a copy; it’s a re-computation of whatever the mutable rows currently say.

It cannot be guaranteed

The original research put it bluntly: “access should fail if an audit trail is not created.” With synthesis there is no separate audit write to succeed or fail. If a field isn’t stamped, or a source row is written best-effort, the event silently never existed, and no one can tell, because there was never a record whose absence would be noticed.

It is lossy by design

This isn’t a backlog to burn down; it’s structural. The deferred kinds contribute nothing; edit history is collapsed to the latest; repeated events and all-but-the-last refusals are unrecoverable. The most audit-relevant question of all (did someone actually open the secret, and how often?) is exactly the one synthesis can’t answer.

Its retention is not its own

The trail lives precisely as long as the underlying business rows live, with no independent retention window and no auditor export. Today PAM doesn’t prune those rows, so the trail reaches back “far enough.” But the moment any data-lifecycle or privacy process prunes a request or lease, the corresponding history is gone and cannot be recovered. Audit retention should be a policy decision, not a side effect of how long operational data happens to stick around.

The crux

Always-on and queryability are things we can keep improving. Immutability, tamper-evidence, and guaranteed capture are not. They are properties of a separate, write-once store, and no amount of iteration on a projection will produce them; a projection is, by definition, a re-reading of mutable data. So the compliance gap is not a maturity gap that closes over time. It closes only when the record stops being synthesized and starts being written.

Beside the alternatives, this is the “Synthesized (built)” column of the four-way comparison in Section 8: failing or partial on every property but always-on.

8The case for a dedicated, non-generic table

The original research made the case on its own merits. Everything above turns that case from a prediction into an observed result: we tried to avoid the table, and the workarounds in Section 6 plus the gaps in Section 7 are what avoiding it cost.

The original reasons still stand

  • Compliance & retention. A privileged-access trail is a compliance artifact with its own retention, append-only and tamper-evident expectations, and auditor export. It shouldn’t inherit operational data’s lifecycle.
  • Always-on, not entitlement-gated. A governance record should exist wherever PAM is sold. (Synthesis gets this right, so a dedicated store must preserve it, not regress it.)
  • Fidelity. Room for the verdict, the comment, the requester-versus-approver distinction, the lease window, and the governing rule: the columns a sentence feed has no room for.
  • Queryability. Filter by collection, cipher, requester, approver, verdict, kind, date, automated-versus-human; drill from a request to its timeline, a cipher to everyone who held it, a rule to its change history.
  • Isolation. Keep governance data off the high-volume general event path and let the audit schema evolve on its own.
  • Best-effort is not good enough. PAM events must be guaranteed to be logged: access should fail if the trail can’t be written. Only a real write can offer that.
Why non-generic: the part that’s easy to get wrong

The organizational pressure is toward a generic audit table serving many purposes. A generic store could, in fairness, be made immutable, append-only, and guaranteed, fixing three of the gaps above. But a generic schema, by definition, cannot carry PAM’s domain columns: requester vs. approver, verdict, comment, lease window, governing rule, the request→decision→lease relationships. Strip those out and you have reinvented the four-column event feed one layer down, with the same fidelity and queryability failures that disqualified the event log in the first place. Add them back in and it is no longer generic; it is a PAM table wearing a generic label.

Genericness trades away exactly the fidelity that makes a privileged-access audit worth having. Hence the recommendation is specific: dedicated and purpose-built for PAM.

We built the synthesized log to avoid a table, and it works well enough to prove we need one: the trail privileged access needs is a written record, not a re-reading of operational data.

9Why not the existing event system?

The first question anyone asks: Bitwarden already has an organization event system, so why build a separate trail at all? We looked at this closely, because reusing it is the cheapest answer if it works.

We do use it, and we always will

PAM events are emitted to the event system as a standing requirement, whatever is decided here. And that system is more than a viewer: it fans events out to Splunk (HEC), Datadog, webhooks, Slack and Teams, with retries, on both cloud and self-hosted (an AMQP bus, RabbitMQ or Azure Service Bus). So broad distribution, including streaming PAM activity into a SIEM, is already handled. A dedicated audit log replaces none of that, and we are not rebuilding it.

What the event system cannot be is the governance audit trail itself. The differences are structural, not missing features. Here is the event log we have set against what a compliant audit log requires:

  The event log we have A compliant audit log
Capture Best-effort: an async queue, so the action still succeeds if the event is dropped Guaranteed: the action fails if the record can’t be written
Immutability An append store, but not write-once Write-once, append-only by design
Tamper-evidence None; no way to prove history wasn’t altered Hash-chained or signed, and verifiable
Completeness A fixed set of event types; new kinds need the events team Every governance action, by design
Fidelity Four columns: timestamp, client, member, event The full record: requester, approver, verdict, comment, lease window, rule
Queryability By scope and date By collection, cipher, requester, approver, verdict, kind and date, with drill-downs
Retention The general event-telemetry lifecycle Independent and policy-driven, with auditor export
Availability Behind the enterprise-events entitlement Always on, wherever PAM is sold

The four-way comparison in Section 8 places these beside the other options. The point here is narrower: the event log we have is telemetry, and telemetry can’t be patched into a compliance record.

Export formats: the gap worth naming

Because export keeps coming up: the event viewer already offers a CSV dump of its four columns, and the integrations target observability destinations (Splunk HEC, Datadog, webhooks) using each destination’s own payload. What neither produces is a full-fidelity record in the standardized audit formats a SIEM or auditor expects:

Format Where it is expected
CEF Common Event Format: the de-facto SIEM interchange (ArcSight and most SIEMs)
LEEF Log Event Extended Format: IBM QRadar
Syslog (RFC 5424) The standard transport envelope, often carrying CEF or LEEF
OCSF Open Cybersecurity Schema Framework: the modern normalized security-event schema
JSON / NDJSON Structured, full-fidelity records for auditor tooling and pipelines

A purpose-built PAM audit log could emit these at full fidelity (every governance field) and with integrity (append-only, and optionally hash-chained or signed for tamper-evidence). The event pipeline can carry the generic event shape to an observability tool; it cannot produce a compliance-grade, full-fidelity audit record in an auditor’s interchange format. That capability is unique to a dedicated store, and it complements the event system rather than duplicating it.

The relationship, stated plainly

Event system and audit log are complementary, not either/or. The event system remains the org-wide activity feed and the distribution pipe, and always carries PAM events. The dedicated audit log is the governance-grade, queryable, exportable record of privileged access. Reusing the event system for that second job was explored and does not work, which is why a separate store is on the table.

10Storage & scope: the CosmosDB proposal

With the case for a dedicated store made (Section 8), the live counter-proposal is a generic audit/event store on Azure CosmosDB, motivated by scale and volume: rather than build a parallel system that invites “why not unify with events?”, build the generic, high-throughput substrate that existing events could eventually migrate onto. It is a shrewd, politically-aware position, and it bundles two decisions that deserve to be judged separately.

What’s already true in the codebase

The org event log already isn’t SQL in cloud: it runs on Azure Table Storage for cloud and EF/Dapper SQL for self-hosted, both behind a single IEventRepository selected at deploy time. Its write path is a best-effort queue (emit → Azure Queue → processor → Table Storage), and it is queried by scope + date only. There is no Cosmos in the stack yet, and PAM’s synthesized audit is MSSQL. Two takeaways: “not SQL for cloud” is existing precedent, and swapping the store behind an interface is already the house pattern.

Two decisions, bundled: judge them apart

Decision The real question Note

Recalibrating on scale: audit is the wrong workload to size for volume

The CosmosDB suggestion is driven by scale and volume, and on that basis it is sound for the high-volume event firehose (every login, view and sync, across every org). But PAM audit is orders of magnitude smaller: human-driven governance actions (requested, approved, activated, revoked). Sizing the audit store for the firehose’s throughput optimizes the wrong workload, and it is exactly why the two shouldn’t be fused. The scale argument is real; it just lands on the event platform, not on PAM’s trail.

The crux: the two workloads have opposite binding constraints

This is why unification is genuinely hard, and it answers the architect’s own question: what differs events from audits? The existing architecture already encodes the split:

  Event log (activity feed) Audit trail (compliance)
Binding constraint Volume: throughput at scale Integrity: nothing lost or altered
Typical volume Very high (all org activity) Low (privileged actions only)
Durability Best-effort (queue → Table Storage) Guaranteed: no silent loss
Mutability Append, effectively disposable Append-only + immutable + tamper-evident
Retention Telemetry lifecycle Policy-driven, independent, exportable
Query Scope + date Many dimensions + drill-downs
Best-fit tool Cosmos / Table Storage (scale) Transactional SQL (integrity + rich query)

Cosmos optimizes the top-left cell; the audit trail’s binding constraints run down the right-hand column. Serving both from one store optimizes for the wrong axis for one of them. You can share a substrate; you cannot relax audit’s guarantees down to the event bar without recreating the synthesized model’s core flaw.

The generic-Cosmos approach, judged fairly

Where it genuinely wins
  • Scale & volume for the unified firehose, its actual purpose. Cosmos is built for high-throughput append, with TTL and a change feed; SQL fights this at volume.
  • Politically survivable, and consistent with precedent (events are already NoSQL-in-cloud). A shipped generic store beats a blocked perfect one.
  • Schema flexibility suits heterogeneous audit payloads: a generic envelope can still carry a typed PAM payload, so the domain fields are at least storable.
  • One well-run platform beats N bespoke per-feature audit tables. Real long-term wisdom.
Where it costs you, especially under a deadline
  • Scope explosion. It turns “ship PAM audit” into “build the org’s unified audit platform on a new datastore, and set expectations to migrate existing events.” Bigger than the dedicated table, not smaller.
  • Guaranteed capture gets harder. PAM’s writes are in MSSQL; a dedicated SQL audit row can be written in the same transaction. A separate Cosmos store makes it a cross-store dual-write; the only route back to a guarantee is a transactional outbox (SQL machinery you keep anyway, plus delivery lag).
  • Queryability. Cosmos favors point / partition reads; governance needs ad-hoc, cross-dimension queries and drill-downs (request → lease → rule). The partition key is a one-way door; rich queries often force a second analytical store.
  • “Compliance ⇒ Cosmos” is a non-sequitur. Tamper-evidence isn’t free in Cosmos, and Azure SQL has a Ledger feature purpose-built for it. Integrity is orthogonal to SQL-vs-Cosmos.
  • New operational surface + self-hosted. Cosmos is Azure-only and net-new to the stack; self-hosted (MSSQL / Postgres / …) needs a SQL backend regardless.

The reconciliation

Recommendation: write now behind the seam, relocate later

The one thing you cannot defer is starting to write a durable record: synthesized has no write to redirect later, so “migrate later” only works if you write now. The IAccessAuditEventRepository seam already exists. For the deadline, ship the cheapest compliant-enough backend behind it: a dedicated PAM audit table in MSSQL, written in the same transaction as the business change. That delivers immutability, guaranteed capture, rich queries and self-hosted support today, and PAM’s low volume makes the in-transaction write a non-issue.

Later, if the generic Cosmos platform is built, repoint the interface, exactly as IEventRepository swaps Table Storage and SQL today. Net: agree with the long-term vision; don’t make PAM the project that builds it.
