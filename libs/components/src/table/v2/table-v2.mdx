# Table V2

`bit-table-v2` is a column-driven evolution of `bit-table`. Each column is a `<bit-column>` wrapper
containing a `<bit-header-cell>` for the header markup and a `*bitCellDef` cell template for the
per-row cell markup. The table renders a CSS-grid layout of ARIA-role elements
(`role="table"`/`"row"`/`"columnheader"`/`"cell"`); the consumer's `<bit-header-cell>`/`<bit-cell>`
become the grid items, and column widths map to grid tracks.

A `TableDef`, bound via `[tableDef]`, supplies the row data and the typed column references — the
one piece that has to be shared so `*bitCellDef` cells are strictly typed. Sort, displayed columns,
loading, filtering, selection, and pagination are configured with inputs on the table and its helper
components.

Use `bit-table-v2` for new tables. The legacy `bit-table` continues to work and isn't being migrated
— see [Table](?path=/docs/component-library-table--docs) for its docs.

## Quick start

```ts
type User = { id: number; name: string; email: string };

class UserListComponent {
  protected readonly table = defineTable<User>(this.users);
  // ...
}
```

```html
<bit-table-v2 [tableDef]="table">
  <bit-column sortable defaultSort="asc">
    <bit-header-cell>Id</bit-header-cell>
    <bit-cell *bitCellDef="table.columns.id; let row">{{ row.id }}</bit-cell>
  </bit-column>
  <bit-column sortable>
    <bit-header-cell>Name</bit-header-cell>
    <bit-cell *bitCellDef="table.columns.name; let row">{{ row.name }}</bit-cell>
  </bit-column>
  <bit-column>
    <bit-header-cell>Email</bit-header-cell>
    <bit-cell *bitCellDef="table.columns.email; let row">{{ row.email }}</bit-cell>
  </bit-column>
</bit-table-v2>
```

`table.columns.X` is a typed column reference: autocomplete lists every property of `User`, typos
fail at compile time, and the row type flows through `*bitCellDef` so `let row` is typed as `User`
inside each cell template.

## The model

`TableDef<T, S>` is the **typed contract**, bound via `[tableDef]`. It carries the row type `T` and
declared synthetic column names `S` (see [Synthetic columns](#synthetic-columns)), plus the row
`data` and the typed `columns` references. It exists for the one thing inputs can't do: thread
`T`/`S` across the `<bit-table-v2>` ↔ `*bitCellDef` boundary so `let row` and the column keys are
strictly typed. (The filter-values type isn't here — it's inferred from the `[filter]` fn.)

```ts
const table = defineTable<User, "actions">(users);
table.columns.email; // typed ColumnRef, drives `*bitCellDef`'s `let row`
```

`data` is a `Signal<T[]>` — the table re-renders as it changes. For mutable data, pass a writable
signal and `.set()` it; for static data, wrap the array in `signal(...)`. There's no `setData` — you
own the signal.

Everything else is configured with **inputs** on the table and its helper components, not on the
model:

| Input / helper                              | Purpose                                                                 |
| ------------------------------------------- | ----------------------------------------------------------------------- |
| `[displayedColumns]`                        | Columns to show, in order. Omit to show all in declaration order.       |
| `[(sort)]`                                  | Active sort (`{ column, direction }`); header clicks cycle it.          |
| `[loading]`                                 | Show skeleton rows in place of data.                                    |
| `[filter]`                                  | Client-side row test `(row, filterValues) => boolean`.                  |
| `[selection]`                               | Selection config (`{ multiple, canSelect }`); `(selectedChange)` emits. |
| `<bit-table-paginator>`                     | Projected; owns page state, slices the rows.                            |
| `table.filterValues()` / `table.filtered()` | Read off a `#t="bitTableV2"` ref for server queries.                    |

## Column definitions

Each `<bit-column>` is a thin wrapper carrying column-level metadata. The column key, the cell
template, and the row context come from a `*bitCellDef` child; the header markup comes from a
`<bit-header-cell>` child.

| Input         | Type                | Notes                                                           |
| ------------- | ------------------- | --------------------------------------------------------------- |
| `sortable`    | `boolean`           | Click-to-sort on the header.                                    |
| `defaultSort` | `"asc"` \| `"desc"` | Initial sort direction. First column with this set wins.        |
| `sortFn`      | `SortFn`            | Custom comparator. See [Sorting](#sorting).                     |
| `width`       | `string`            | Grid track size (e.g. `"120px"`, `"1fr"`). Defaults to `"1fr"`. |

### `*bitCellDef` — the cell template

The structural directive `*bitCellDef` captures the per-row cell template and identifies which
column it belongs to. Its required input is a typed column reference from the model
(`table.columns.*`).

```html
<bit-column>
  <bit-cell *bitCellDef="table.columns.email; let row">{{ row.email }}</bit-cell>
</bit-column>
```

`let row` is typed as `User` via `ngTemplateContextGuard` — `row.emial` fails to compile.

### `<bit-header-cell>` — the header template

The header markup is a plain `<bit-header-cell>` child of the column. No `*bitCellDef`-style
directive — the column reads the cell's template directly, and the column key already lives on
`*bitCellDef`.

```html
<bit-column sortable>
  <bit-header-cell><i class="bwi bwi-envelope tw-me-1"></i> Email</bit-header-cell>
  <bit-cell *bitCellDef="table.columns.email; let row">{{ row.email }}</bit-cell>
</bit-column>
```

The sort button affordance is added by `<bit-header-cell>` automatically when the surrounding
`<bit-column>` is sortable. Click toggles direction; aria-sort and aria-pressed are wired for you.

### Synthetic columns

For columns that don't correspond to a field on `T` (action columns, expand/collapse, etc.), declare
them via the model's second type parameter, then reference them like any other column:

```ts
const table = defineTable<User, "actions">(users);
```

```html
<bit-column>
  <bit-header-cell></bit-header-cell>
  <bit-cell *bitCellDef="table.columns.actions; let row">
    <button bitIconButton="bwi-ellipsis-v">Actions</button>
  </bit-cell>
</bit-column>
```

`table.columns.actions` still carries the row type, so `let row` is typed `User`.

### Reordering and hiding columns

`[displayedColumns]` is the ordered list of columns to show; a column is rendered only if it's
listed, at the position it's listed — names omitted aren't rendered even though they're declared.
**Omit it to show every column in declaration order.** Bind a signal to reorder or hide at runtime:

```html
<bit-table-v2 [tableDef]="table" [displayedColumns]="['name', 'id']">
  <bit-column>
    <bit-header-cell>Id</bit-header-cell>
    <bit-cell *bitCellDef="table.columns.id; let row">{{ row.id }}</bit-cell>
  </bit-column>
  <bit-column>
    <bit-header-cell>Name</bit-header-cell>
    <bit-cell *bitCellDef="table.columns.name; let row">{{ row.name }}</bit-cell>
  </bit-column>
  <bit-column>
    <bit-header-cell>Email (not displayed)</bit-header-cell>
    <bit-cell *bitCellDef="table.columns.email; let row">{{ row.email }}</bit-cell>
  </bit-column>
</bit-table-v2>
```

### Composing columns into reusable wrappers

Columns register via DI lookup of the ancestor `<bit-table-v2>`, so a `<bit-column>` inside another
component's view still finds the table. Pass the model in for its typed refs:

```ts
@Component({
  selector: "demo-status-column",
  imports: [BitColumnComponent, BitCellDefDirective, BitHeaderCellComponent, BitCellComponent],
  template: `
    <bit-column sortable>
      <bit-header-cell>Status</bit-header-cell>
      <bit-cell *bitCellDef="table().columns.status; let row">
        <span class="…">{{ row.status }}</span>
      </bit-cell>
    </bit-column>
  `,
})
class DemoStatusColumnComponent {
  readonly table = input.required<TableDef<User>>();
}
```

Used as:

```html
<bit-table-v2 [tableDef]="table">
  <bit-column>
    <bit-header-cell>Name</bit-header-cell>
    <bit-cell *bitCellDef="table.columns.name; let row">{{ row.name }}</bit-cell>
  </bit-column>
  <demo-status-column [table]="table" />
</bit-table-v2>
```

## Cell content layout

`<bit-cell>` provides a slot vocabulary for rich cells — a leading icon or tile (`slot=start`), the
title (default slot), a subtitle (`slot=secondary`), and a trailing affordance (`slot=end`). Plain
cells use only the default slot and the layout collapses to bare content.

```html
<bit-column sortable defaultSort="asc">
  <bit-header-cell>Name</bit-header-cell>
  <bit-cell *bitCellDef="table.columns.name; let row">
    <bit-icon-tile slot="start" icon="bwi-globe" size="sm" />
    {{ row.name }}
    <span slot="secondary">{{ row.email }}</span>
  </bit-cell>
</bit-column>
```

The `truncate` input (default `true`) controls whether title and subtitle truncate with an ellipsis
or wrap. Slot=secondary and slot=end are hidden when empty, so plain cells render exactly the
default slot's content with no empty wrappers.

## Empty state

When no rows render (in column-def mode) — empty data, or a filter that excluded everything — the
table shows a default `<bit-no-items>` below the header. Project `slot="empty"` to override it:

```html
<bit-table-v2 [tableDef]="table">
  <div slot="empty" class="tw-p-10 tw-text-center tw-text-muted">No items found</div>
  <bit-column>…</bit-column>
</bit-table-v2>
```

Manual mode has no empty state — you control the rows.

## Loading

Bind `[loading]` (e.g. a resource's `isLoading()`) to show skeleton rows in place of data. Control
the count with `[loadingRows]` (default 3). Each column renders its optional `bitCellLoading`
template; columns without one get a default skeleton. Loading takes precedence over the empty state.

```html
<bit-table-v2 [tableDef]="table" [loading]="usersResource.isLoading()" [loadingRows]="5">
  <bit-column>
    <bit-header-cell>Name</bit-header-cell>
    <bit-cell *bitCellDef="table.columns.name; let row">{{ row.name }}</bit-cell>
    <bit-cell *bitCellLoading><bit-skeleton-text class="tw-w-3/4" /></bit-cell>
  </bit-column>
</bit-table-v2>
```

## Manual mode

For simple presentational tables — no sort, filter, selection, or virtualization — skip the model
and `<bit-column>`, and project rows directly:

```html
<bit-table-v2>
  <bit-header-row>
    <bit-header-cell>Id</bit-header-cell>
    <bit-header-cell>Name</bit-header-cell>
  </bit-header-row>
  @for (r of items(); track r.id) {
  <bit-row>
    <bit-cell>{{ r.id }}</bit-cell>
    <bit-cell>{{ r.name }}</bit-cell>
  </bit-row>
  }
</bit-table-v2>
```

The consumer projects one `<bit-header-row>` and any number of `<bit-row>` components. The header
row picks up the design's styling automatically. Manual mode has no `[tableDef]` — the consumer
iterates their own data directly; columns default to equal-share (`grid-auto-columns: 1fr`).

Mode selection is automatic: any projected `<bit-column>` ⇒ column-def mode; otherwise the
consumer's `<bit-header-row>` and `<bit-row>` components render directly. Don't mix the two within a
single table — if you need any of the framework features (sort, selection, etc.), use column-def.

## Sorting

`sortable` on `<bit-column>` enables click-to-sort. The cycle is unsorted → ascending → descending →
ascending. Apply an initial sort with `defaultSort`:

```html
<bit-column sortable defaultSort="asc">
  <bit-header-cell>Name</bit-header-cell>
  <bit-cell *bitCellDef="table.columns.name; let row">{{ row.name }}</bit-cell>
</bit-column>
```

For non-trivial comparisons, pass a `sortFn` — return the ascending comparison; the table applies
the active direction:

```ts
sortByName = (a: User, b: User) => a.name.localeCompare(b.name);
```

```html
<bit-column sortable [sortFn]="sortByName">
  <bit-header-cell>Name</bit-header-cell>
  <bit-cell *bitCellDef="table.columns.name; let row">{{ row.name }}</bit-cell>
</bit-column>
```

Sort is a two-way input: bind `[(sort)]` to a `SortState` signal (`{ column, direction }`) to read
it or set it programmatically (e.g. to persist/restore). Header clicks cycle it; the comparator
comes from the column's `sortFn`, so the bound state stays serializable.

```html
<bit-table-v2 [tableDef]="table" [(sort)]="sort">…</bit-table-v2>
```

## Filtering

Filtering works like a reactive form. Filter chips (`bit-filter-chip` / `bit-filter-toggle`)
projected into the toolbar each declare a `key` and own their selection; the `bitTableFilter` bridge
registers them, and the table collects their values into **`table.filterValues()`** — a
`{ key: value }` object, like a `FormGroup`'s `.value`. You pass one **`[filter]`** fn
`(row, values)`; the table derives the rendered rows, select-all scope, paginator count, and a "no
matches" empty state from it. **Annotate the fn's `values` parameter** with your filter shape — the
table infers it and types `filterValues()` to match. Seed initial values with `[filters]`. No
per-chip state, no `ngModel`.

```ts
type Filters = { type?: ItemType; favorites?: boolean };

const table = defineTable<Member, "actions">(members);
filter = (row: Member, f: Filters) =>
  (!f.type || row.type === f.type) && (!f.favorites || row.favorite);
```

```html
<bit-table-v2 [tableDef]="table" [filter]="filter" [filters]="{ type: 'login' }">
  <bit-table-toolbar>
    <bit-filter-chip key="type" placeholderText="Type" nullLabel="All" bitTableFilter>
      <bit-filter-option [value]="'login'">Login</bit-filter-option>
    </bit-filter-chip>
    <bit-filter-toggle
      key="favorites"
      label="Favorites"
      icon="bwi-star"
      bitTableFilter
    ></bit-filter-toggle>
  </bit-table-toolbar>
  <bit-column>...</bit-column>
</bit-table-v2>
```

The chip owns its menu UI and value; the bridge wires it to the table via the `FILTER_HOST` token,
so the chip never depends on the table type and works outside a table too. For **server-side**
filtering, omit `[filter]` and read `filterValues()` off a `#t="bitTableV2"` template reference to
build your query (feed pre-filtered rows to `data`). See the
[Filter Menu](?path=/docs/component-library-filter-menu--docs) for composing the chips (sections,
in-menu search, and a radio + checkbox group in one chip, which nests under the chip's key).

### Search

A `<bit-search>` projected into the table (typically the toolbar) is adopted automatically as a
filter under the `search` key — no `bitTableFilter`, no `ngModel`. Its term lands in
`filterValues().search`; read it in your `filter` alongside the chip keys:

```ts
filter: (row, f) =>
  (!f.search || row.name.toLowerCase().includes(f.search.toLowerCase())) &&
  (!f.type || row.type === f.type),
```

## Toolbar

`<bit-table-toolbar>`, projected into the table, renders inside the chrome above the header row. It
lays out in two rows:

- **Top row** — the `<bit-search>` (capped at ~400px in its slot) and any `slot="end"` controls
  (buttons, etc., right-aligned).
- **Filter row** — the filter chips, with a live "N items" count of the filtered rows at the end.
  Group related chips with a `<bit-filter-divider>`. This row is the unit that collapses into a
  single trigger + dialog on small screens (responsive view).

The toolbar holds no filter state of its own: chips register with the table via `bitTableFilter`,
and a projected `<bit-search>` is auto-adopted as the `search` filter (see [Search](#search) above).

```html
<bit-table-v2 [tableDef]="table">
  <bit-table-toolbar>
    <bit-search class="tw-flex-1" placeholder="Search"></bit-search>
    <button slot="end" bitButton buttonType="primary">New</button>
    <bit-filter-chip key="type" placeholderText="Type" bitTableFilter>...</bit-filter-chip>
    <bit-filter-divider></bit-filter-divider>
    <bit-filter-toggle
      key="favorites"
      label="Favorites"
      icon="bwi-star"
      bitTableFilter
    ></bit-filter-toggle>
  </bit-table-toolbar>
  <bit-column>...</bit-column>
</bit-table-v2>
```

## Selection

Pass a `[selection]` config and the table prepends an internal checkbox column (generated by the
table; consumers don't declare it). Listen to `(selectedChange)` for the selected rows:

```html
<bit-table-v2
  [tableDef]="table"
  [selection]="{ multiple: true }"
  (selectedChange)="onSelected($event)"
>
  …columns…
</bit-table-v2>
```

The header checkbox targets the filtered rows (matches CDK conventions). Selected rows pick up a
soft background. (Pass a **stable** config reference — a field or getter, not an inline literal — so
the selection isn't rebuilt on every change.)

### Selectable rows

When only some rows are selectable, supply a `canSelect` predicate. The table renders a checkbox
only for selectable rows, and select-all / indeterminate scope to them; the predicate is also
enforced when selecting, so a non-selectable row can't be selected even programmatically.

```html
<bit-table-v2 [tableDef]="table" [selection]="{ multiple: true, canSelect: canManage }"
  >…</bit-table-v2
>
```

### Bulk actions

`<bit-bulk-actions-bar>` projected inside the table reads `selectedCount` implicitly from the
selection via DI; its clear button also clears the selection. No explicit wiring:

```html
<bit-table-v2 [tableDef]="table">
  <bit-bulk-actions-bar>
    <bit-bulk-action [action]="onDelete" icon="bwi-trash" label="Delete" />
    <bit-bulk-action [action]="onArchive" icon="bwi-archive" label="Archive" />
  </bit-bulk-actions-bar>

  …columns…
</bit-table-v2>
```

Outside a table, the bar's explicit API (`[selectedCount]`, `(clear)`) still works — see
[Bulk Actions Bar](?path=/docs/component-library-bulk-actions-bar--docs).

## Pagination

Project a `<bit-table-paginator>` and the table slices its filtered (and sorted) rows to its page.
The paginator owns the page state (`[(pageIndex)]`, `[pageSize]`, `[pageSizeOptions]`) and reads the
total back from the table; its footer shows the current row range, a page-size select, prev/next
controls, and a page input:

```html
<bit-table-v2 [tableDef]="table">
  …columns…
  <bit-table-paginator [pageSize]="10" [pageSizeOptions]="[10, 25, 50, 100]" />
</bit-table-v2>
```

<Canvas of={stories.Pagination} />

By default pagination is **client-side**: the table slices its filtered (and sorted) rows to the
current page, and the total ("of N") is the filtered row count. Bind `[(pageIndex)]` /
`[(pageSize)]` to persist or restore state.

For **server-side** pagination, give the paginator an explicit `[length]` (the total row count).
That selects server mode: the table does not slice — `data()` is expected to already hold only the
current page (fetch it keyed off the paginator's `pageIndex` / `pageSize`):

```html
<bit-table-paginator [length]="totalCount" [(pageIndex)]="page" [pageSize]="10" />
```

A paginator and [virtual scrolling](#virtual-scrolling) are mutually exclusive — virtualization
already renders large sets efficiently.

## URL sync

Bind **`[queryParam]`** to a namespace and the table mirrors its filters, sort, and pagination to
the URL query string under that prefix — so a filtered/sorted/paged view is shareable and survives a
reload. On load the params restore state (winning over `[filters]` and `defaultSort`); on change the
table writes back with `replaceUrl` (flat history) and `merge` (params it doesn't own are left
alone). Inactive/default facets leave no param behind, so the URL self-cleans.

```html
<bit-table-v2 [tableDef]="table" [filter]="filter" queryParam="items">…</bit-table-v2>
<!-- ?items.type=login&items.favorite=true&items.sort=name&items.direction=asc&items.page=2 -->
```

<Canvas of={stories.UrlSync} />

Each filter chip syncs under its own `key`; sort, direction, page, and pageSize use those keys
within the namespace (so don't give a chip one of those keys). Values are read back by shape —
`true`/`false` → boolean, integers → number, a repeated key → array — which is lossy at the edges (a
literal `"true"`, a numeric-string id), the trade for readable URLs. A distinct prefix per table
lets several sync on one page. URL sync needs a router in context; without one (e.g. a dialog) it's
a no-op.

Under the hood this is **`queryParamStore`** (in `@bitwarden/components`), a generic writable signal
mirrored to a URL namespace — reach for it directly to sync any component-owned state (a tab, a
search term) to the query string, table or not.

## Scrolling and the sticky header

By default the table grows to fit its rows and scrolls with the page. To bound it instead — so the
body scrolls and the header row stays pinned — give it a height in one of two ways:

- **`[maxHeight]`** (any CSS length) — the table is as tall as its content, up to this cap, then
  scrolls.
- **`fill`** — the table grows to fill its container's height. Use it inside a bounded, full-height
  region such as the body of a [`bit-page`](?path=/docs/component-library-layout-page--docs), which
  is the idiomatic way to make a table fill the main content area:

```html
<bit-page>
  <h1 slot="header" bitTypography="h1">Members</h1>
  <bit-table-v2 [tableDef]="table" fill>…columns…</bit-table-v2>
</bit-page>
```

`bit-page` owns the page's layout — a fixed header/footer and a fill body that scrolls — so the
table stays out of the layout business; it just fills the box it's given. Reach for `maxHeight` when
the table is one bounded block among others (a dialog, a card); reach for `fill` + `bit-page` when
the table is the page.

## Virtual scrolling

For large datasets, set `[virtualRowHeight]` (the height per row, in px). The table renders inside a
`cdk-virtual-scroll-viewport`; only the visible rows are stamped. Virtualization scrolls the table
itself, so it needs a bounded height — **`[maxHeight]` or `fill`**:

```html
<bit-table-v2 [tableDef]="table" [virtualRowHeight]="64" maxHeight="400px" [trackBy]="trackById">
  …columns…
</bit-table-v2>
```

`[virtualRowHeight]` lays rows out with a fixed grid — virtual scrolling needs predictable row
geometry. Specify `width="…"` on columns for stable widths.

`[trackBy]` is optional but recommended for stable identity across data updates.

## Type safety

`BitTableV2Component<T>` is generic over the row type, inferred from `[tableDef]`. The
`TableDef<T>`'s `table.columns.X` references (fields and synthetics alike) carry `T` to
`*bitCellDef`, which uses `ngTemplateContextGuard` to type `let row` as `T`. Result:

- `table.columns.email.typed` fails (no such property)
- `table.columns.emial` fails (typo)
- `let row` inside `*bitCellDef` is typed `T` — `row.email` autocompletes, `row.emial` fails

This is meaningfully stronger than `MatTable`, which provides only static-API typing (cell `let-row`
is `any`).

## Accessibility

- Always include a header — even an empty one — for each column; screen readers use it to
  contextualize cell data.
- Avoid spanning data across cells.
- For repeating row-level actions, give each one an associating label — `aria-label="Edit Netflix"`
  rather than a bare "Edit".
- The select-all checkbox is labelled by the `selectAllRows` i18n key; row checkboxes use
  `selectRow`. Override via the cell template if richer context is needed.
- Sort buttons expose `aria-sort` (`"ascending"` / `"descending"`) on the `<th>` and `aria-pressed`
  on the button.
