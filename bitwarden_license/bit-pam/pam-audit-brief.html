Privileged Access Management · Governance

The PAM Audit Log

What the synthesized trail is, why it isn’t a compliance-grade record, and the lowest-effort way to fix that without blowing the deadline.

Status: Discussion draft (condensed) Date: 1 July 2026 Long form: pam-audit.html
Bottom line

The synthesized audit log is a fine in-product view, but it is a projection of live data, not a record: immutability, tamper-evidence, and guaranteed capture cannot be added to a projection. The fix that fits the deadline is to start writing audit events now: behind the interface that already exists (IAccessAuditEventRepository), into a dedicated MSSQL table committed in the same transaction as the action, then repoint that interface at a generic / Cosmos platform later if one gets built. Agree with the long-term generic vision; don’t make PAM the project that builds it.

1The situation

PAM lets a member request temporary, approved access to a protected credential, hold it for a fixed window, and lose it when the window closes. The lifecycle the log has to describe: a request is submitted, approved or denied, activated into a time-boxed lease, used to open the secret, and finally expired or revoked.

Governance needs to answer afterwards: who requested access to what, who decided, who opened the secret, and when did it end? That is the audit log. Unlike an activity feed, a privileged-access trail is expected to be complete, immutable, tamper-evident, and guaranteed to exist.

A dedicated PAM audit table was declined, not on technical merit but because a new table is only welcome if it is generic enough to serve many purposes. So today the trail is synthesized at query time from data the PAM entities already keep. No new table, no stored audit records.

2Why the synthesized log isn’t a record

Because nothing is stored, the trail is reconstructed on demand from the same live rows that normal operations edit and delete. It covers the request and lease lifecycle and rule administration; it cannot capture credential opens, full rule-edit history, or the kill-switch / freeze; those are deferred and simply don’t appear.

The rule it runs on

An event can be reconstructed only if it left a durable, distinct, single-occurrence mark on a surviving row. Break any of those and it can’t be synthesized at all.

The four ways it breaks, and the compromise each forced

How synthesis breaks What we had to do
Repeated events
credential opens 		Defer them: one row can’t remember N opens. They contribute nothing today. A per-open serialized list on the lease would capture them.
Overwrites
rule edits		 Keep only the latest edit; prior versions and who made them are lost.
Hard deletes
rule removed		 Switch delete to soft-delete and hide the row from every active read, a real behaviour change.
Refused attempts
activation rejected 		Add a field to leave a mark, but it records only the last refusal.

Every fix is either “capture less” or “bend the model to preserve a mark”: effort spent approximating what a written record would give for free.

The gap is structural, not a backlog

Three things an auditor expects (immutable, tamper-evident, guaranteed capture) cannot be added to a projection. They are properties of a separate, write-once store. No amount of iteration on a re-reading of mutable data produces them; the gap closes only when the record is written instead of derived.

What an audit record needs Synthesized trail (today)
Immutable / append-only ✗ it is the mutable business data
Tamper-evident ✗ no independent copy to check against
Guaranteed capture ✗ no separate write to fail on
Complete ✗ deferred kinds absent; edit / access history lost
Independent retention & export ✗ dies with the source rows
Always-on (no entitlement) ✓ rides on PAM’s own data
Rich queryability ● partial, limited to what rows carry

3Dedicated vs generic / CosmosDB

The live counter-proposal is a generic audit/event store on CosmosDB, motivated by scale. It bundles two separable decisions (generic-vs-PAM-specific, and Cosmos-vs-SQL), and the Cosmos rationale is contingent on the first: it is right for the high-volume event firehose (every login, view and sync), but PAM audit is orders of magnitude smaller. Events and audits pull in opposite directions:

  Event log (activity feed) Audit trail (compliance)
Binding constraint Volume: throughput at scale Integrity: nothing lost or altered
Durability Best-effort (queue → Table Storage) Guaranteed: no silent loss
Mutability Append, disposable Immutable, tamper-evident
Query Scope + date Many dimensions + drill-downs
Best-fit tool Cosmos / Table Storage (scale) Transactional SQL (integrity)

So the scale argument is real; it just lands on the event platform, not PAM’s trail. Judged fairly:

Where generic / Cosmos wins
  • Scale for the firehose, its actual purpose.
  • Politically survivable, and matches precedent (events are already NoSQL-in-cloud).
  • Schema flexibility for heterogeneous payloads.
  • One platform beats N bespoke per-feature tables.
Where it costs you, under a deadline
  • Scope explosion: “ship PAM audit” becomes “build the org’s audit platform.”
  • Guaranteed capture gets harder: a separate store is a dual-write needing an outbox.
  • Queryability: point/partition reads vs governance drill-downs.
  • New surface + self-hosted needs a SQL backend regardless.

On guaranteed capture specifically: PAM’s writes are already in MSSQL, so a dedicated audit row commits in the same transaction; moving it to Cosmos is what turns that into a cross-store dual-write. And the swap-later path already exists: the event log runs on Table Storage (cloud) and SQL (self-host) behind one IEventRepository, chosen at deploy time, with no Cosmos in the stack yet.

The reconciliation

The one thing you cannot defer is starting to write a durable record: synthesized has no write to redirect later, so “migrate later” only works if you write now. Ship the dedicated MSSQL table behind the existing seam for the deadline; repoint it at the generic platform later, exactly as IEventRepository already swaps backends.

4Open questions

  • Compliance targets. Do SOC 2 / ISO 27001 / HIPAA actually require immutable, tamper-evident privileged-access logs? That decides whether Section 2 is “must-fix” or “nice to have.”
  • Best-effort or guaranteed? Do we hold the line on “access fails if it can’t be logged”? The synthesized model can’t offer it.
  • Credential-open auditing. Who opened the secret, and how often, is the one genuinely high-volume PAM stream; if required, it could justify a Cosmos-class store as a separate stream while governance actions stay transactional.
  • Retention & export independent of operational-data lifecycle.
  • Ownership boundary with the team that owns the existing event log: parallel emission stands, but who owns the governance-grade store?
  • Audience & the “generic-only” constraint: can it be revisited given the Section 3 argument?
