# Table V2

`bit-table-v2` is a column-driven evolution of `bit-table`. Each column is a `<bit-column>` wrapper
containing two structural templates — `*bitColumnHeader` for the header markup and `*bitColumnFor`
for the per-row cell markup — and column-level metadata (sortable, sortFn, defaultSort, width) lives
as inputs on the wrapper. The table renders a CSS-grid layout of ARIA-role elements
(`role="table"`/`"row"`/`"columnheader"`/`"cell"`); the consumer's `<bit-header-cell>`/`<bit-cell>`
become the grid items, and column widths map to grid tracks.

Use `bit-table-v2` for new tables. The legacy `bit-table` continues to work and isn't being migrated
— see [Table](?path=/docs/component-library-table--docs) for its docs.

<Primary />

## Quick start

```ts
type User = { id: number; name: string; email: string };

class UserListComponent {
  protected readonly users = new TableDataSource<User>();
  protected readonly displayedColumns = ["id", "name", "email"];
  // ...
}
```

```html
<bit-table-v2 [dataSource]="users" [displayedColumns]="displayedColumns">
  <bit-column sortable defaultSort="asc">
    <bit-header-cell *bitColumnHeader>Id</bit-header-cell>
    <bit-cell *bitColumnFor="users.columns.id; let cell">{{ cell.id }}</bit-cell>
  </bit-column>
  <bit-column sortable>
    <bit-header-cell *bitColumnHeader>Name</bit-header-cell>
    <bit-cell *bitColumnFor="users.columns.name; let cell">{{ cell.name }}</bit-cell>
  </bit-column>
  <bit-column>
    <bit-header-cell *bitColumnHeader>Email</bit-header-cell>
    <bit-cell *bitColumnFor="users.columns.email; let cell">{{ cell.email }}</bit-cell>
  </bit-column>
</bit-table-v2>
```

`users.columns.X` is a typed reference: autocomplete lists every property of `User`, typos fail at
compile time, and the row type flows through `*bitColumnFor` so `let cell` is typed as `User` inside
each cell template.

## Column definitions

Each `<bit-column>` is a thin wrapper carrying column-level metadata. The column key, the cell
template, and the row context come from a `*bitColumnFor` child; the header markup comes from a
`*bitColumnHeader` child.

| Input         | Type                | Notes                                                           |
| ------------- | ------------------- | --------------------------------------------------------------- |
| `sortable`    | `boolean`           | Click-to-sort on the header.                                    |
| `defaultSort` | `"asc"` \| `"desc"` | Initial sort direction. First column with this set wins.        |
| `sortFn`      | `SortFn`            | Custom comparator. See [Sorting](#sorting).                     |
| `width`       | `string`            | Grid track size (e.g. `"120px"`, `"1fr"`). Defaults to `"1fr"`. |

### `*bitColumnFor` — the cell template

The structural directive `*bitColumnFor` captures the per-row cell template and identifies which
column it belongs to. Its required input is a typed column reference from the data source.

```html
<bit-column>
  <bit-cell *bitColumnFor="users.columns.email; let cell">{{ cell.email }}</bit-cell>
</bit-column>
```

`let cell` is typed as `User` via `ngTemplateContextGuard` — `cell.emial` fails to compile.

### `*bitColumnHeader` — the header template

A separate structural directive captures the header. No inputs — the column key already lives on
`*bitColumnFor`.

```html
<bit-column sortable>
  <bit-header-cell *bitColumnHeader><i class="bwi bwi-envelope tw-me-1"></i> Email</bit-header-cell>
  <bit-cell *bitColumnFor="users.columns.email; let cell">{{ cell.email }}</bit-cell>
</bit-column>
```

The sort button affordance is added by `<bit-header-cell>` automatically when the surrounding
`<bit-column>` is sortable. Click toggles direction; aria-sort and aria-pressed are wired for you.

### Synthetic columns

For columns that don't correspond to a field on `T` (action columns, expand/collapse, etc.), use
`synthetic()`:

```html
<bit-column>
  <bit-header-cell *bitColumnHeader></bit-header-cell>
  <bit-cell *bitColumnFor="users.synthetic('actions'); let cell">
    <button bitIconButton="bwi-ellipsis-v">Actions</button>
  </bit-cell>
</bit-column>
```

`synthetic()` still carries the row type, so `let cell` is typed `User`.

### Reordering and hiding columns

Columns appear in `[displayedColumns]` order. Columns whose names aren't listed don't render, even
if declared:

```html
<bit-table-v2 [dataSource]="users" [displayedColumns]="['name', 'id']">
  <bit-column>
    <bit-header-cell *bitColumnHeader>Id</bit-header-cell>
    <bit-cell *bitColumnFor="users.columns.id; let cell">{{ cell.id }}</bit-cell>
  </bit-column>
  <bit-column>
    <bit-header-cell *bitColumnHeader>Name</bit-header-cell>
    <bit-cell *bitColumnFor="users.columns.name; let cell">{{ cell.name }}</bit-cell>
  </bit-column>
  <bit-column>
    <bit-header-cell *bitColumnHeader>Email (not displayed)</bit-header-cell>
    <bit-cell *bitColumnFor="users.columns.email; let cell">{{ cell.email }}</bit-cell>
  </bit-column>
</bit-table-v2>
```

### Composing columns into reusable wrappers

Columns register via DI lookup of the ancestor `<bit-table-v2>`, so a `<bit-column>` inside another
component's view still finds the table:

```ts
@Component({
  selector: "demo-status-column",
  imports: [
    BitColumnComponent,
    BitColumnHeaderDirective,
    BitColumnForDirective,
    BitHeaderCellComponent,
    BitCellComponent,
  ],
  template: `
    <bit-column sortable>
      <bit-header-cell *bitColumnHeader>Status</bit-header-cell>
      <bit-cell *bitColumnFor="ds().columns.status; let cell">
        <span class="…">{{ cell.status }}</span>
      </bit-cell>
    </bit-column>
  `,
})
class DemoStatusColumnComponent {
  readonly ds = input.required<TableDataSource<User>>();
}
```

Used as:

```html
<bit-table-v2 [dataSource]="users" [displayedColumns]="['name', 'status']">
  <bit-column>
    <bit-header-cell *bitColumnHeader>Name</bit-header-cell>
    <bit-cell *bitColumnFor="users.columns.name; let cell">{{ cell.name }}</bit-cell>
  </bit-column>
  <demo-status-column [ds]="users" />
</bit-table-v2>
```

## Cell content layout

`<bit-cell>` provides a slot vocabulary for rich cells — a leading icon or tile (`slot=start`), the
title (default slot), a subtitle (`slot=secondary`), and a trailing affordance (`slot=end`). Plain
cells use only the default slot and the layout collapses to bare content.

```html
<bit-column sortable defaultSort="asc">
  <bit-header-cell *bitColumnHeader>Name</bit-header-cell>
  <bit-cell *bitColumnFor="users.columns.name; let cell">
    <bit-icon-tile slot="start" icon="bwi-globe" size="sm" />
    {{ cell.name }}
    <span slot="secondary">{{ cell.email }}</span>
  </bit-cell>
</bit-column>
```

The `truncate` input (default `true`) controls whether title and subtitle truncate with an ellipsis
or wrap. Slot=secondary and slot=end are hidden when empty, so plain cells render exactly the
default slot's content with no empty wrappers.

## Manual mode

For simple presentational tables — no sort, filter, selection, or virtualization — skip
`<bit-column>` and project rows directly:

```html
<bit-table-v2>
  <bit-header-row>
    <bit-header-cell>Id</bit-header-cell>
    <bit-header-cell>Name</bit-header-cell>
  </bit-header-row>
  @for (r of items(); track r.id) {
  <bit-row>
    <bit-cell>{{ r.id }}</bit-cell>
    <bit-cell>{{ r.name }}</bit-cell>
  </bit-row>
  }
</bit-table-v2>
```

The consumer projects one `<bit-header-row>` and any number of `<bit-row>` components. The header
row picks up the design's styling (secondary background, bottom border) automatically.

Manual mode has no `[dataSource]` plumbing — the consumer iterates their own data signal/observable
directly. Rich cells still work; `<bit-cell>`'s slot vocab is the same in both modes. Columns
default to equal-share (`grid-auto-columns: 1fr`) since no `<bit-column>` registry exists to drive
`grid-template-columns`.

Mode selection is automatic: any projected `<bit-column>` ⇒ column-def mode; otherwise the
consumer's `<bit-header-row>` and `<bit-row>` components render directly. Don't mix the two within a
single table — if you need any of the framework features (sort, selection, etc.), use column-def.

## Sorting

`sortable` on `<bit-column>` enables click-to-sort. The cycle is unsorted → ascending → descending →
ascending. Apply an initial sort with `defaultSort`:

```html
<bit-column sortable defaultSort="asc">
  <bit-header-cell *bitColumnHeader>Name</bit-header-cell>
  <bit-cell *bitColumnFor="users.columns.name; let cell">{{ cell.name }}</bit-cell>
</bit-column>
```

For non-trivial comparisons, pass a `sortFn`:

```ts
sortByName = (a: User, b: User, direction?: SortDirection) => {
  const result = a.name.localeCompare(b.name);
  return direction === "asc" ? result : -result;
};
```

```html
<bit-column sortable [sortFn]="sortByName">
  <bit-header-cell *bitColumnHeader>Name</bit-header-cell>
  <bit-cell *bitColumnFor="users.columns.name; let cell">{{ cell.name }}</bit-cell>
</bit-column>
```

Sort state lives on the `TableDataSource` — see
[Table → Sorting](?path=/docs/component-library-table--docs#sorting) for the mechanics.

## Filtering

Apply via `TableDataSource.filter`:

```ts
dataSource.filter = "search value";
// or
dataSource.filter = (row) => row.orgType === "family";
```

See [Table → Filtering](?path=/docs/component-library-table--docs#filtering).

## Selection

Pass a `SelectionModel<T>` and the table prepends an internal checkbox column (generated by the
table; consumers don't declare it):

```ts
selection = new SelectionModel<User>(true, []);
```

```html
<bit-table-v2
  [dataSource]="users"
  [selection]="selection"
  [displayedColumns]="['id', 'name', 'email']"
>
  …columns…
</bit-table-v2>
```

The header checkbox targets `dataSource.filteredData` (matches CDK conventions). Selected rows pick
up a soft background via the existing `bitRow` styling. The model is consumer-owned — the table
reads from and writes to it but does not construct it.

### Bulk actions

`<bit-bulk-actions-bar>` projected inside the table reads `selectedCount` implicitly from the
table's selection model via DI; its clear button also clears the model. No explicit wiring:

```html
<bit-table-v2 [dataSource]="users" [selection]="selection" [displayedColumns]="cols">
  <bit-bulk-actions-bar>
    <bit-bulk-action [action]="onDelete" icon="bwi-trash" label="Delete" />
    <bit-bulk-action [action]="onArchive" icon="bwi-archive" label="Archive" />
  </bit-bulk-actions-bar>

  …columns…
</bit-table-v2>
```

Outside a table, the bar's explicit API (`[selectedCount]`, `(clear)`) still works — see
[Bulk Actions Bar](?path=/docs/component-library-bulk-actions-bar--docs).

## Virtual scrolling

For large datasets, set `[rowSize]` (the height per row, in px). The table renders inside a
`cdk-virtual-scroll-viewport`; only the visible rows are stamped:

```html
<bit-table-v2
  [dataSource]="users"
  [displayedColumns]="['id', 'name']"
  [rowSize]="64"
  [trackBy]="trackById"
>
  …columns…
</bit-table-v2>
```

`[rowSize]` forces `table-fixed` layout regardless of `[layout]` — virtual scrolling needs
predictable row geometry. Specify `width="…"` on columns for stable widths.

`[trackBy]` is optional but recommended for stable identity across data updates.

## Type safety

`BitTableV2Component<T>` is generic over the row type, inferred from `[dataSource]`. The
`users.columns.X` and `users.synthetic(X)` references carry `T` to `*bitColumnFor`, which uses
`ngTemplateContextGuard` to type `let-cell` as `T`. Result:

- `users.columns.email.typed` fails (no such property)
- `users.columns.emial` fails (typo)
- `let cell` inside `*bitColumnFor` is typed `T` — `cell.email` autocompletes, `cell.emial` fails

This is meaningfully stronger than `MatTable`, which provides only static-API typing (cell `let-row`
is `any`).

## Accessibility

- Always include a header — even an empty one — for each column; screen readers use it to
  contextualize cell data.
- Avoid spanning data across cells.
- For repeating row-level actions, give each one an associating label — `aria-label="Edit Netflix"`
  rather than a bare "Edit".
- The select-all checkbox uses `aria-label="Select all rows"`; row checkboxes use
  `aria-label="Select row"`. Override via the cell template if richer context is needed.
- Sort buttons expose `aria-sort` (`"ascending"` / `"descending"`) on the `<th>` and `aria-pressed`
  on the button.
